## 前言

你给 GPT-4 一段 prompt，它能写出漂亮的代码。你让它调用工具、记住上下文、自主规划步骤、出错时自我修正——它做不到了。

从 LLM 到 Agent 的跨越，不是换一个更大的模型，而是在模型外面搭一层工程系统。这层系统通常叫 **Agent Harness**（Agent 脚手架）——它负责把 LLM 的「语言能力」变成 Agent 的「行动能力」。

这篇文章是 Agent Harness 工程系列的上篇，聚焦编排框架的核心架构：执行循环怎么设计？工具怎么集成？记忆怎么管理？多个 Agent 怎么协作？读完你会对 Agent 系统的骨架有一个完整的认识。

下篇会聊评测体系和工程化实践——怎么衡量 Agent 的能力，怎么让它在生产中可靠运行。

## 1. 什么是 Agent Harness

### 1.1 一个类比：赛车与车手

LLM 像一个天赋异禀的车手——引擎强劲、反应极快，但它被绑在方向盘上，看不见路况，也摸不到油门。

Agent Harness 就是那辆赛车：方向盘、油门、刹车、仪表盘、GPS、无线电——所有让车手真正能跑起来的基础设施。没有这辆车，车手再强也只能原地空转。

```text
┌─────────────────────────────────────────┐
│              Agent Harness              │
│  ┌───────┐ ┌──────┐ ┌───────┐ ┌─────┐ │
│  │ 执行  │ │ 工具 │ │ 记忆  │ │ 规划│ │
│  │ 循环  │ │ 集成 │ │ 管理  │ │ 模块│ │
│  └───┬───┘ └──┬───┘ └───┬───┘ └──┬──┘ │
│      └────────┴─────────┴────────┘     │
│                    │                    │
│              ┌─────┴─────┐             │
│              │   LLM     │             │
│              └───────────┘             │
└─────────────────────────────────────────┘
```

### 1.2 为什么需要 Harness

LLM 本身有几个根本限制：

**无状态。** 每次调用都是独立的。上一次对话的上下文不会自动带进来。

**无手脚。** 它只能生成文本，不能执行代码、搜索网页、读写文件。

**无自我监控。** 它不知道自己什么时候犯了错，也不知道什么时候该停下来重新思考。

**无协作能力。** 单个 LLM 无法把任务分配给不同的「专家」，也无法管理多个推理线程。

Harness 的作用就是补齐这些缺失。它不是一个模型，而是一套围绕模型运行的工程系统。

## 2. 执行循环：Agent 的心脏

### 2.1 ReAct 模式

几乎所有 Agent 框架的核心都是一个**执行循环（execution loop）**。最经典的模式是 ReAct（Reasoning + Acting），由 Yao 等人在 2022 年提出。

每一轮循环做三件事：

```text
while not done:
    thought = llm.reason(context)      # 思考：当前情况是什么，下一步该做什么
    action  = llm.decide(thought)      # 行动：调用工具或生成输出
    observation = execute(action)      # 观察：工具返回了什么
    context.update(thought, action, observation)
```

这个循环看起来简单，但它定义了 Agent 的基本行为模式：**感知→推理→行动→观察→再推理**。

> 📄 **ReAct: Synergizing Reasoning and Acting in Language Models**
> Yao, Zhao, Yu, et al. ICLR 2023.
> 提出 ReAct 框架，将推理（Chain-of-Thought）和行动（Tool Use）统一在一个循环中。

### 2.2 循环的控制流

实际工程中，执行循环远比上面的伪代码复杂。需要考虑：

**终止条件。** Agent 什么时候停下来？是得到了最终答案，还是达到了最大步数，还是用户主动中断？没有好的终止策略，Agent 可能陷入死循环。

**错误恢复。** 工具调用失败了怎么办？是重试、换一种方式、还是直接告诉用户？

**预算控制。** 每一步都消耗 token 和 API 调用。需要设置 token 预算和步数上限，防止失控。

**状态持久化。** 如果 Agent 需要运行很长时间（比如几分钟甚至几小时），中间状态怎么保存？进程崩溃了怎么恢复？

这些看似琐碎的工程细节，恰恰是 Agent 从 demo 走向生产的关键。

### 2.3 主流框架的执行循环

不同框架对执行循环的实现各有特色：

| 框架 | 循环模式 | 特点 |
|------|----------|------|
| **LangGraph** | 状态图（StateGraph） | 用有向图定义执行流，支持条件分支和循环 |
| **AutoGen** | 对话驱动 | 多 Agent 通过对话协议协作，循环嵌入在对话中 |
| **CrewAI** | 任务编排 | 预定义任务和角色，按顺序或并行执行 |
| **OpenAI Assistants** | 托管循环 | 平台管理执行循环，开发者只定义工具和指令 |
| **OpenAI Agents SDK** | Handoff 协议 | 生产级多 Agent SDK，支持 handoff、guardrails 和 tracing |

LangGraph 的设计值得多说几句。它把 Agent 的执行流程建模为一个**有向图**：节点是计算步骤（LLM 调用、工具执行、条件判断），边是控制流。这让开发者可以精确控制循环的每一步，包括在哪里分支、在哪里合并、在哪里允许人工介入。

## 3. 工具集成：让 Agent 有手有脚

### 3.1 工具调用的基本范式

LLM 本身只能生成文本。要让它「做事」，需要一层工具集成。

基本流程是：

1. LLM 生成一个**工具调用请求**（tool call），包含工具名和参数
2. Harness 解析这个请求，执行对应的函数
3. 把执行结果作为**观察（observation）** 返回给 LLM
4. LLM 根据结果决定下一步

```text
LLM: "我需要搜索最新的论文"
  → tool_call: web_search(query="LLM agent harness 2026")
Harness: 执行搜索，返回结果
  → observation: "找到 5 篇相关论文：..."
LLM: "让我看第 3 篇的摘要"
  → tool_call: fetch_paper(id=3)
```

### 3.2 工具定义的标准化

工具定义通常包含三部分：

```python
tool = {
    "name": "web_search",
    "description": "搜索互联网获取最新信息",
    "parameters": {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "搜索关键词"},
            "num_results": {"type": "integer", "default": 5}
        },
        "required": ["query"]
    }
}
```

关键在 `description` 字段。LLM 是通过阅读描述来决定是否使用工具的——描述写得不好，LLM 就会用错工具或者根本不用。这其实是一种**面向 LLM 的 API 设计**，和面向人类的 API 设计有相通之处，但也有独特的考量。

### 3.3 工具编排的挑战

当工具数量增多，问题就来了。

**选择困难。** 如果 Agent 有 50 个工具可用，LLM 经常选错。解决方案包括：动态筛选（根据当前任务只暴露相关工具）、工具分组（把相关工具组织成子 Agent）、分层工具（先选类别再选具体工具）。

**参数幻觉。** LLM 可能编造不存在的参数值。比如搜索时编一个不存在的日期格式，或者调用数据库时传入错误的字段名。需要在 Harness 层做参数校验和类型检查。

**错误传播。** 一个工具调用失败，错误信息需要足够清晰，让 LLM 理解失败原因并调整策略。直接返回 stack trace 没用，需要把错误翻译成 LLM 能理解的自然语言。

## 4. 记忆管理：Agent 的长期认知

### 4.1 为什么需要记忆

LLM 的上下文窗口是有限的。即使 128K token 的窗口，也装不下一个持续运行数天的 Agent 的全部历史。

更重要的是，Agent 需要**选择性记忆**——不是所有信息都同等重要。和人类一样，Agent 需要记住关键决策和教训，忘记无关细节。

### 4.2 记忆的分类

Agent 的记忆通常分为三类：

**工作记忆（Working Memory）。** 当前任务的上下文——最近几轮对话、正在处理的中间结果。通常存在上下文窗口内，是最直接但也最受限的记忆。

**短期记忆（Short-term Memory）。** 当前会话的完整历史。超出上下文窗口的部分需要做摘要或检索。常见做法是用一个滑动窗口，保留最近 N 轮对话的原文，更早的对话压缩成摘要。

**长期记忆（Long-term Memory）。** 跨会话持久化的知识——用户偏好、项目约定、过去学到的教训。通常存储在向量数据库或结构化存储中，通过检索增强（RAG）在需要时加载。

```text
┌──────────────────────────────────────────┐
│              记忆层次                     │
├──────────┬──────────┬────────────────────┤
│ 工作记忆  │ 短期记忆  │     长期记忆       │
│ (上下文)  │ (会话)   │   (跨会话持久化)   │
├──────────┼──────────┼────────────────────┤
│ 最近 N 轮 │ 完整对话  │ 向量数据库 /       │
│ 当前任务  │ 历史摘要  │ 结构化存储 /       │
│ 中间结果  │          │ 知识图谱           │
└──────────┴──────────┴────────────────────┘
```

### 4.3 记忆的工程实现

**向量存储。** 最常用的长期记忆方案。把过去的对话、学到的知识编码成向量，存入 Chroma、Pinecone 或 FAISS。需要时通过语义检索召回相关记忆。

**摘要压缩。** 当对话历史超出窗口时，用 LLM 把旧对话压缩成摘要。关键挑战是压缩时不能丢失重要信息——一个看似无关的闲聊可能包含后续任务的关键线索。

**结构化记忆。** 不是所有记忆都适合用向量表示。用户偏好、项目规则、决策历史更适合结构化存储（JSON、数据库表）。MemGPT 的做法是把记忆组织成类似操作系统的虚拟内存结构，有「主存」和「外存」的分层。

> 📄 **MemGPT: Towards LLMs as Operating Systems**
> Packer, Wooders, Lin, et al. 2023.
> 提出把 LLM 的记忆管理类比操作系统的内存层次结构，实现分层记忆和自主记忆管理。

### 4.4 记忆的检索策略

存了记忆不等于能用好。检索策略直接影响 Agent 的表现：

**Top-K 检索。** 最简单——按相似度取前 K 条。但可能漏掉关键信息，也可能引入噪声。

**时间衰减。** 近期记忆权重更高。这符合直觉：昨天学到的教训比一个月前的更相关。

**重要性加权。** 给记忆打重要性分数，重要的记忆更容易被召回。可以手动标注（用户标记关键信息），也可以自动评估（用另一个 LLM 判断重要性）。

**混合检索。** 结合向量相似度和关键词匹配。纯向量检索可能错过精确匹配的信息，混合方案更鲁棒。

## 5. 多 Agent 编排：从独奏到交响

### 5.1 为什么需要多 Agent

单个 Agent 做所有事情，很快就会遇到瓶颈：

**上下文污染。** 一个 Agent 同时处理多个子任务，上下文窗口被各种无关信息占满，推理质量下降。

**角色冲突。** 同一个 Agent 既要写代码又要审代码，容易陷入「自己审自己」的盲区。

**并行效率。** 很多任务可以并行执行，单个 Agent 只能串行处理。

多 Agent 系统的思路是：让不同的 Agent 扮演不同角色，各自专注自己的任务，通过通信协议协作。

### 5.2 编排模式

多 Agent 的编排有几种常见模式：

**层级式（Hierarchical）。** 一个「管理者」Agent 负责分配任务，多个「执行者」Agent 各自完成子任务。管理者汇总结果，决定下一步。

```text
          ┌──────────┐
          │ Manager  │
          └────┬─────┘
       ┌───────┼───────┐
       ▼       ▼       ▼
   ┌──────┐ ┌──────┐ ┌──────┐
   │Coder │ │Review│ │Test  │
   └──────┘ └──────┘ └──────┘
```

**对话式（Conversational）。** 多个 Agent 在一个共享的对话空间中交流，像人类团队开会一样。AutoGen 的核心设计就是这种模式。

**流水线式（Pipeline）。** 任务按固定顺序经过多个 Agent，每个 Agent 完成一个阶段。类似工厂的流水线。

**竞争式（Competitive）。** 多个 Agent 独立解决同一个问题，最后由一个评判 Agent 选择最优方案。这类似于「思维锦标赛」，可以提高答案质量，但代价是更多的计算资源。

### 5.3 通信协议

Agent 之间怎么交流，是多 Agent 系统的核心设计问题。

**共享黑板（Blackboard）。** 所有 Agent 读写同一个共享状态。简单直接，但需要处理并发冲突。

**消息传递（Message Passing）。** Agent 之间通过显式消息通信。更灵活，但需要设计消息格式和路由规则。

**事件驱动（Event-Driven）。** Agent 发布事件，其他 Agent 订阅感兴趣的事件。松耦合，适合异步场景。

### 5.4 工程挑战

多 Agent 系统带来了一些独特的工程问题：

**调试困难。** 多个 Agent 的交互行为难以预测和复现。一个 Agent 的输出可能因为另一个 Agent 的微小变化而完全不同。需要完善的日志和追踪机制。

**成本控制。** 每个 Agent 的每一步都消耗 token。5 个 Agent 各跑 10 步，就是 50 次 LLM 调用。需要精细的预算分配和成本监控。

**一致性保证。** 多个 Agent 对同一个实体的理解可能不一致。比如一个 Agent 认为文件路径是 `/src/main.py`，另一个认为是 `src/main.py`。需要共享的上下文管理。

> 📄 **AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation**
> Wu, Yen, Liu, et al. 2023.
> 提出多 Agent 对话框架，让多个 LLM Agent 通过对话协作解决复杂任务。

> 📄 **MetaGPT: Meta Programming for A Multi-Agent Collaborative Framework**
> Hong, Zhuge, Chen, et al. ICLR 2024.
> 将标准化操作流程（SOP）引入多 Agent 系统，用流水线范式减少协作中的级联幻觉。

## 6. 规划与反思：Agent 的高阶能力

### 6.1 任务分解

复杂任务不能一步完成，需要分解成子任务。

**层级规划（Hierarchical Planning）。** 先生成高层计划（「先调研、再设计、再实现、再测试」），再逐步细化每一步。

**树搜索（Tree of Thoughts）。** 把规划建模为搜索问题——每个节点是一个可能的中间状态，用 BFS 或 DFS 探索最优路径。Yao 等人在 2023 年的 Tree of Thoughts 论文中系统探索了这个方向。

**反思式规划。** 执行过程中不断回顾和调整计划。发现某条路走不通，就回溯到上一个决策点，换一条路。

### 6.2 自我反思

自我反思（Self-Reflection）是让 Agent 从错误中学习的关键能力。

**Reflexion 框架。** Shinn 等人在 2023 年提出，Agent 在每次失败后生成一段文字反思（「我失败了，原因是 X，下次应该 Y」），存入记忆，下次遇到类似任务时参考。

**自我批评。** 让 Agent 审视自己的输出，找出问题并修正。这可以用同一个 LLM 做（角色切换），也可以用不同的 LLM 做（增加多样性）。

**外部反馈。** 通过工具调用获取客观反馈——代码能不能跑、测试通不通过、编译有没有报错。这比 LLM 的自我评估更可靠。

> 📄 **Reflexion: Language Agents with Verbal Reinforcement Learning**
> Shinn, Cassano, Gopinath, et al. NeurIPS 2023.
> 提出 Reflexion 框架，让 Agent 通过语言形式的自我反思从失败中学习。

### 6.3 规划与执行的交织

在实际系统中，规划和执行不是分开的两个阶段，而是紧密交织的。

一个好的 Agent 会：

- 先做一个粗略的计划
- 开始执行，同时观察环境反馈
- 根据反馈调整计划
- 遇到意外时重新规划
- 完成任务后回顾整个过程

这种「规划→执行→反思→再规划」的循环，是 Agent Harness 中最有价值的模式之一。

## 7. 调度与编排：循环之外的定时任务

在讨论 Agent 的重复行为时，有一个容易混淆的概念需要厘清：**循环（Loop）**和**定时任务（Scheduled Task / Cron Job）**。它们表面上都涉及「重复执行」，但在 Harness 架构中扮演着完全不同的角色。

### 7.1 核心区别

**循环是任务内部的迭代。** 一次任务执行中，Agent 反复执行「思考→行动→观察」直到完成。循环的驱动力是任务本身的复杂度——任务没完成，循环就继续。

**定时任务是任务外部的触发。** 一个调度器在固定时间或固定间隔启动一次独立的任务执行。每次触发都是一个全新的任务，和上一次执行没有上下文关联。

```text
循环（Loop）—— 一次任务内部的迭代：
┌────────────────────────────────────────┐
│  Task: "修复这个 bug"                    │
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │Round1│→│Round2│→│Round3│→ 完成 ✓   │
│  └──────┘  └──────┘  └──────┘          │
└────────────────────────────────────────┘

定时任务（Cron）—— 多次独立的任务触发：
┌──────┐        ┌──────┐        ┌──────┐
│Task A│        │Task B│        │Task C│
│9:00  │  ...   │9:05  │  ...   │9:10  │
│无关联│        │无关联│        │无关联│
└──────┘        └──────┘        └──────┘
```

### 7.2 对比总结

| 维度 | 循环（Loop） | 定时任务（Cron） |
|------|-------------|-----------------|
| **触发方式** | 任务未完成自动继续 | 外部调度器按时间触发 |
| **上下文** | 每轮共享上下文，累积信息 | 每次执行相互独立 |
| **终止条件** | 任务完成 / 达到上限 | 时间到了就结束 |
| **目的** | 迭代收敛到一个好的结果 | 定期执行重复性工作 |
| **典型场景** | 调试代码、搜索信息、生成-验证-修复 | 定时巡检、日报生成、数据同步 |
| **状态** | 有状态（中间结果跨轮传递） | 无状态（或依赖外部持久化） |

### 7.3 组合使用

在实际系统中，循环和定时任务经常组合使用：

**定时任务触发循环。** 一个 cron job 每天凌晨 2 点启动「检查系统健康」任务，这个任务内部使用循环来逐个检查各项指标。

**循环中嵌入定时等待。** Agent 在循环中遇到需要等待的外部条件（如等待 CI 结果），可以暂停循环，设置定时回调，等条件满足后恢复。

**定时任务间的隐式数据流。** 多个定时任务通过共享的外部存储（数据库、文件）形成隐式的数据流。任务 A 写入数据，任务 B 定时读取并处理——这本质上是一个跨进程的「流水线」，但每一环都是独立的定时任务。

区分这两种机制很重要，因为它决定了你该在哪里处理状态、在哪里做错误恢复、以及如何控制成本。把定时任务当成循环来设计，会导致不必要的资源浪费；把循环当成定时任务来设计，则会丢失迭代积累的认知。

## 8. 长期任务：邮件、跨会话记忆与异步等待

前面的讨论大多围绕单次会话内的 Agent 行为展开。但 Harness 真正展现架构价值的地方，是那些跨越单次对话边界的长期任务。

### 8.1 邮件：不只是一次性的读和回

处理一封邮件看起来是单次任务——读邮件、理解意图、生成回复、发送。但真实的邮件工作流远比这复杂：

**多轮协商。** 你收到一封会议邀请，需要确认时间。Agent 检查你的日历，发现有冲突，提议新时间，对方回复又调整，如此反复直到达成一致。这不是一个「读→回」的单次操作，而是一个跨越数小时甚至数天的协商过程。

```text
收到邀请 → 检查日历 → 发现冲突 → 提议新时间
    ↑                                    │
    └──── 对方回复调整 ← 等待对方确认 ←──┘
```

**邮件链跟踪。** 一个项目讨论可能跨越十几封邮件。Agent 需要在每次收到新邮件时，回溯整个对话链，理解上下文演变，判断是否需要自己行动。这是一种增量理解——每封新邮件都是对整个故事的更新。

**批量处理中的优先级排序。** 早上打开邮箱有 50 封未读邮件。Agent 不是按时间顺序逐一处理，而是先扫描全部，按紧急程度和重要性排序，优先处理需要立即回复的。这个「扫描→分类→处理」的流程，是 Harness 层的调度能力，不是循环内部的事。

**定时与异步的交织。** 邮件场景完美展示了调度与记忆的组合：定时任务每 15 分钟检查新邮件（外部触发），收到新邮件后启动循环来处理（内部迭代）。处理过程中可能还需要等待对方回复（异步等待），形成更复杂的嵌套结构。

### 8.2 跨会话记忆：连接断裂的对话

单次对话中的 Agent 有一个隐含假设：所有上下文都在当前窗口内。但很多真实任务的信息分布在多次对话中。

**跨对话的知识延续。** 你昨天让 Agent 调试了一个 CSS 问题，今天又遇到类似的。如果 Harness 提供了长期记忆能力，Agent 可以检索之前的解决方案，尝试应用，发现不完全适用后调整方案，成功后更新记忆。

**记忆的维护。** 长期记忆不是静态存储——它需要维护。被频繁引用的记忆得到强化，长期未被使用的逐渐衰减，过时的被标记或归档，矛盾的触发冲突解决。这个过程类似于人类大脑的遗忘曲线，是 Harness 层需要提供的持久化能力。

**知识的蒸馏。** 原始记忆是零散的对话片段。通过周期性的蒸馏，Agent 可以把多条相关记忆合并为结构化知识：

```text
原始记忆 1: "用户偏好 TypeScript 而不是 JavaScript"
原始记忆 2: "用户的项目使用 pnpm 而不是 npm"
原始记忆 3: "用户喜欢简洁的代码风格"
    ↓ 蒸馏
结构化知识: "用户画像——TypeScript + pnpm + 简洁风格"
```

### 8.3 设计启示

邮件和跨会话记忆揭示了 Harness 层在处理长期任务时几个重要的架构原则：

**持久化状态。** 跨越对话边界的任务必须把中间状态存储到外部（数据库、文件系统），而不是依赖上下文窗口。进程崩溃、会话结束后，任务应该能从上次中断的地方恢复。

**异步友好。** 长期任务中的很多步骤需要等待外部事件（对方回复邮件、CI 构建完成、API 限流解除）。Harness 需要支持「暂停→等待→恢复」的模式，而不是忙等待。

**增量更新。** 长期记忆不需要每次全量加载。Harness 的每一步只需要检索与当前上下文相关的记忆子集，避免上下文爆炸。

**优雅降级。** 长期任务可能运行数天甚至数周。当记忆库增长到很大时，Harness 应该能优雅降级——减少检索范围、降低更新频率、压缩旧记忆——而不是崩溃。

## 结语

Agent Harness 的编排层，本质上是在回答一个问题：**怎样把 LLM 的语言能力，变成在真实世界中可靠行动的能力？**

执行循环给了 Agent 行动的节奏，工具集成给了它手脚，记忆管理给了它认知，多 Agent 编排给了它协作能力，规划和反思给了它自我改进的可能。调度与编排让 Harness 能处理循环之外的定时任务和异步事件，长期任务的状态管理让 Agent 的能力跨越单次对话的边界。

这些组件单独看都不复杂，但组合在一起，就构成了一个远比单个 LLM 强大的系统。

而在这个系统之上，还有一层更抽象的设计——Loop Engineering（循环工程）。如果说 Harness 是提供工具和基础设施的「赛道」，Loop 就是在赛道上跑的「跑法」：怎么设计迭代模式、怎么让循环自动收敛、怎么编排多个循环并行工作。Harness 提供能力，Loop 编排行为。

不过，能跑起来只是第一步。更大的挑战是：怎么知道这个 Agent 到底做得好不好？怎么保证它在各种场景下都可靠？怎么在成本和质量之间找到平衡？

这些是下篇要聊的内容——Agent Harness 的评测体系和工程化实践。
