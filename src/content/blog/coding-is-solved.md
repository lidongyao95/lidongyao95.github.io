## 前言

想象一个场景：你是一个创业公司的 CTO，团队有 8 个工程师。过去两年，你们的产品路线图永远被「人手不够」卡着——想做的功能排到了明年，技术债还不完，新功能上线永远比计划晚三个月。

然后有一天，你的一个工程师告诉你：他昨天一个人做了 150 个 PR。

不是 150 行代码，是 150 个 Pull Request。而且他大部分时间不是在写代码，而是在手机上开着一堆 session，让 AI Agent 并行地写代码、跑测试、修 CI、处理代码审查。他做的只是定目标、拆任务、看结果。

这不是虚构。这是 Anthropic 的 Boris Cherny——Claude Code 的负责人——在 Sequoia Capital 的访谈中描述的真实工作方式。他管这叫「coding is solved」。

这四个字在技术圈引发了巨大的争议。有人觉得这是夸大其词，有人觉得这是未来已来，有人觉得这只是在卖产品。但如果你仔细听 Boris 到底在说什么，会发现他真正在讨论的不是「代码不重要了」，而是一个更深层的问题：**当代码实现变成了一种通用能力，真正稀缺的东西是什么？**

这篇文章试着拆解这个问题的答案。

## 1. 「Coding is Solved」到底在说什么

### 1.1 不是所有代码都被解决了

先说 Boris 没有说的。他没有说所有软件工程问题都消失了。他明确承认：超大型复杂代码库、冷门语言、特殊系统、深重的历史包袱——这些仍然没有被完全解决。

他说的是：在**很多主流技术栈**里，模型已经足够可靠地承担几乎全部编码工作。

他的例子是 Claude Code 自身。代码库主要是 TypeScript 和 React——都是模型非常熟悉的「分布内」技术。到后来，他个人基本不手写代码，而是让 Claude Code 写 100% 的代码。他有一天做了大约 150 个 PR，专门测试个人产能的上限。

这和我们在 [Loop Engineering](/blog/loop-engineering) 中讨论的范式转移是一回事：开发者从「写代码的人」变成「设计循环的人」。Boris 的实践只是把这个理念推到了极致——不是偶尔让 AI 帮忙，而是彻底把编码工作交给 Agent。

### 1.2 等下一代模型

更值得注意的是 Boris 对那些「还没解决」的场景的态度。当被问到冷门语言、超大型代码库怎么办时，他的回答本质上是一句话：**等下一代模型。**

这不是逃避问题。这是一个经验判断——过去两年，模型能力的提升速度和可靠性提升，已经多次让「不可能」变成了「还凑合」再变成「挺好的」。Boris 的 Claude Code 团队自己也经历过这个过程：产品上线的前 6 个月体验很一般，他自己只让 Claude 写大概 10% 的代码。直到下一代模型成熟，体验才真正起飞。

这种「等模型」的心态，本身就是一种新的工程判断力。过去，工程师需要评估「这个技术方案行不行」；现在，你还需要评估「当前的模型能力行不行，以及多久后就行了」。

## 2. 一个人的 Agent 军团

### 2.1 手机上的开发者

Boris 描述的工作方式，可能是我见过的对「AI-native 开发者」最极致的描绘。

他现在大量工作甚至是在手机上完成的。Claude 应用里有一个 code tab，他在那里开多个 session，每个 session 再启动多个 Agent。他的日常工作流长这样：

```text
┌─────────────────────────────────────────┐
│           Boris 的工作日                 │
│                                         │
│  手机 / 笔记本                           │
│    ├── Session 1: 功能 A 开发             │
│    │    └── 3 agents (code/test/review)  │
│    ├── Session 2: bug 修复               │
│    │    └── 2 agents (diagnose/fix)      │
│    ├── Session 3: 重构任务               │
│    │    └── 4 agents (analyze/refactor)  │
│    ├── ...                              │
│    └── Session N: 定时任务               │
│         └── /loop: 监控 PR / CI / Twitter│
│                                         │
│  夜间: 几千个 agent 做深度工作            │
└─────────────────────────────────────────┘
```

同时跑 5 到 10 个 session，背后可能有几百个 agent。晚上让几千个 agent 做更深的工作。用 `/loop` 让 Claude 通过定时任务重复执行——自动照看 PR、修 CI、rebase、处理 flaky test。甚至定时抓 Twitter 上的用户反馈并聚类总结。

### 2.2 从「写代码」到「调度 Agent」

这个工作流的核心转变是：Boris 的角色不再是「写代码的人」，甚至不是「审代码的人」。他是一个 **Agent 调度者**——设定目标、组织任务、审查结果、持续调度。

这和我们讨论的 Loop Engineering 的六块积木（Osmani, 2026）完美对应：

| Osmani 的积木 | Boris 的实践 |
|--------------|-------------|
| **Automations** | `/loop` 定时任务、CI 自动修复、PR 自动照看 |
| **Worktrees** | 每个 session 隔离的工作空间 |
| **Skills** | Claude Code 自身的项目知识文件 |
| **Plugins** | GitHub、CI/CD、Twitter API 集成 |
| **Sub-agents** | code/test/review 分离的多 agent 架构 |
| **Memory** | 跨 session 的上下文传递和状态持久化 |

Boris 不是在理论上讨论这些概念——他是每天这样工作的人。这可能是目前对 Loop Engineering 最有说服力的真实案例。

### 2.3 关键洞察：并行化

Boris 反复强调的一个词是**并行**。传统开发是串行的——一个工程师一次只能做一件事，做完才能开始下一件。但 Agent 可以并行——你可以同时启动十几个任务，每个任务由一组 Agent 独立完成。

这意味着个人产能的上限被彻底改写了。过去，一个人一天能做 3-5 个 PR 就已经是高产出了。现在，一个人在手机上调度几百个 agent，一天可以做上百个 PR。瓶颈不再是编码速度，而是你能不能同时想清楚这么多任务该怎么做。

## 3. 软件开发的三层能力模型

从 Boris 的访谈中，可以提炼出一个清晰的三层能力模型。这不是他原话，但我觉得这是他整个论述的底层逻辑：

```text
┌────────────────────────────────────────────┐
│  第 1 层：知道要做什么                       │
│  领域知识 · 产品判断 · 业务理解               │
├────────────────────────────────────────────┤
│  第 2 层：知道如何组织 AI 做事                │
│  拆任务 · 并行化 · 审查 · 循环 · 自动化       │
├────────────────────────────────────────────┤
│  第 3 层：知道如何让组织适应                  │
│  让 PM/设计/数据/运营都能参与软件构建          │
└────────────────────────────────────────────┘
```

### 3.1 第一层：领域知识成为核心竞争力

Boris 用了一个精彩的类比：**印刷术**。

印刷术降低了书籍的生产成本，推动了识字率的上升。过去只有抄写员能生产书籍，印刷术之后，人人都能出版。但出版变得容易之后，真正重要的不再是「你能不能印一本书」，而是「你有没有值得写的东西」。

软件 Agent 正在做同样的事。它降低了软件的生产成本，让「构建软件」成为更普遍的能力。但代码变容易之后，真正重要的不再是「你能不能写代码」，而是「你懂不懂你要解决的问题」。

Boris 的关键判断是：**未来最适合写会计软件的人，可能不是传统工程师，而是非常懂会计的人。**

这句话如果成立，它意味着软件开发的中心正在发生一次根本性的位移——从「程序员把业务需求翻译成代码」，转向「领域专家直接借助 AI 表达和实现自己的系统」。

### 3.2 第二层：Agent 编排能力

即使你有了领域知识，你还是需要知道怎么把想法变成可执行的系统。这就是第二层能力：**组织 AI 做事的能力**。

这包括：把一个大目标拆成可并行的子任务、设计验证标准让 Agent 知道自己做得对不对、设置循环让 Agent 自动迭代和收敛、在 Agent 失败时知道怎么调整策略。

这就是 Loop Engineering 的核心——也是 Boris 在 Claude Code 团队里身体力行的工作方式。他不是在写代码，他是在设计一个让代码被自动写出来的系统。

### 3.3 第三层：组织适配

最容易被忽视的是第三层。技术变了，但如果组织不变，效率提升会被组织摩擦吃掉。

Boris 认为未来会出现更多真正跨学科的 **generalist**。过去说工程通才，通常指一个工程师同时会前端、后端、iOS、服务端。但未来的通才是**跨职能**的：工程师懂产品和设计，设计师能写代码，PM 能写代码，数据科学家、用户研究、财务人员也能直接改代码或构建内部工具。

他举 Anthropic 自己团队为例：Claude Code 团队里，工程经理、产品经理、设计师、数据科学家、财务、用户研究人员都写代码。每个人仍然有自己的专业判断力，但「写软件」正在变成共同能力——就像「用电脑」在 30 年前是专业技能，现在是基本素养。

## 4. 对 SaaS 和创业的影响

### 4.1 SaaS 不会死，但护城河在变

Sequoia 的主持人问了一个尖锐的问题：如果写代码的成本下降 10 倍甚至 100 倍，会不会出现 SaaS 末日？

Boris 的回答比较克制。他不认为所有 SaaS 都会被瞬间摧毁，因为某些护城河仍然有效：**网络效应**（你的用户在你的平台上）和**规模经济**（大公司的基础设施成本优势）。

但一些传统的护城河确实在变弱：

| 护城河类型 | 变化趋势 | 原因 |
|-----------|---------|------|
| 网络效应 | 基本不变 | AI 不改变用户之间的关系 |
| 规模经济 | 基本不变 | 基础设施成本优势仍在 |
| 切换成本 | 变弱 | AI 可以帮助迁移数据和流程 |
| 流程壁垒 | 变弱 | 模型越来越擅长理解和复刻流程 |
| 过程优势 | 变弱 | 依赖复杂工作流形成的领先会被压缩 |

### 4.2 创业的爆发

Boris 真正看好的不是大公司的衰落，而是**创业公司的爆发**。他估计未来 10 年能挑战大公司的创业公司数量可能增加 10 倍。

原因很直接：创业公司没有旧流程、旧组织、旧培训和内部阻力。它们可以从第一天就以 AI-native 的方式构建产品和公司。一个 3 人的 AI-native 创业团队，可能拥有过去 30 人团队的产出能力。

这对创业者是一个巨大的结构性利好。过去，很多好的创业想法死在「没人写代码」上——创始人有想法、懂市场，但找不到足够的工程师。现在，这个瓶颈正在消失。

## 5. 产品机会：不只是单次问答

### 5.1 Agentic 产品的方向

被问到「模型变强后什么产品会更有趣」，Boris 提到几个方向：Claude Design（设计工具）、Claude Code 的新功能、**loop / batch / 大规模并行 agent**、以及 computer use。

换句话说，他看好的不是单次问答工具（问一个问题、得一个答案），而是能**持续运行、能并行、能接工具、能自主推进任务**的 agentic 产品。

这和我们对 Agent 产品演进的判断一致。单次问答是 prompt engineering 时代的产品形态；循环迭代是 loop engineering 时代的产品形态。产品的核心交互从「人输入 → 模型输出」变成了「人设定目标 → 系统持续运转 → 人审查结果」。

### 5.2 MCP 和 Computer Use：连接一切

关于 Agent 怎么和外部世界交互，Boris 的观点是分层兜底：

```text
优先级 1：MCP / API
  └── Salesforce、Google Docs、Calendar 等工具
       通过 MCP 或 API 暴露给模型

优先级 2：Computer Use
  └── 没有 API 的软件？模型直接操作电脑界面
       现在还慢，但已经能做很多事
```

他的一个有趣的观点是：对模型来说，MCP、API 和屏幕操作本质上没有区别——都是可处理的 token 和工具调用。模型不关心背后是哪种连接方式，它只关心能不能获取信息、能不能执行动作。

这意味着**所有软件**最终都会成为 Agent 的操作对象——不管它有没有开放 API。对软件厂商来说，这既是机会（让你的产品成为 Agent 生态的一部分），也是威胁（Agent 可以绕过你直接操作你的产品）。

### 5.3 云端还是本地？模型自己决定

关于未来 AI 是跑在云端还是本地，Boris 的回答出人意料地务实：这最终可能不再是工程师手动决定的问题。模型会自己判断——这个任务需要低延迟就用本地模型，需要强推理就用云端模型，需要访问特定环境就用远程容器。

开发者的工作不是选择部署方式，而是确保所有选项都可用，让模型（或 Agent 系统）自己做出最优选择。

## 6. Anthropic 内部的秘密：不是模型，是组织

有人问了一个很多人好奇的问题：Anthropic 内部是不是比外部用户提前用了更强的模型？

Boris 的回答是：模型侧差距不大。他们也使用外部用户能接触到的模型或类似版本，重点是 dogfooding（自己吃自己的狗粮）。

**真正的差距在组织方式。** Anthropic 内部把 Claude 用到几乎所有流程里——SQL 查询、代码开发、内部沟通、agent 之间通过 Slack 协作。不是「我们有秘密技术」，而是「我们的组织流程先适应了 AI-native 工作方式」。

这个观点很值得品味。它暗示了一个不太直觉的结论：**在 AI 时代，组织适配速度可能比技术领先更重要。** 你不需要等最强的模型出来才开始用——用当前最好的模型，配合一个真正适配 AI 工作流的组织，就已经能产生巨大的效率优势。

这和 [Agent Harness 工程](/blog/agent-harness-part1) 中的核心论点一脉相承：模型是引擎，但让车跑起来的是底盘、变速箱和转向系统。Harness 是那些让模型能力变成可靠生产力的工程架构。Boris 的实践证明，这个架构的价值可能比模型本身更大。

## 结语

回到开头的问题：当代码不再是瓶颈，什么才是？

Boris 的访谈给出了一个三层答案：**领域知识**（你得懂你要解决的问题）、**Agent 编排能力**（你得知道怎么组织 AI 来解决问题）、**组织适配**（你得让整个团队都能参与 AI-native 的工作方式）。

「Coding is solved」这句话之所以引起这么大的反响，不是因为它在技术上多么惊人——很多人在过去两年已经逐渐感受到了这个趋势。真正让人震动的是它的**社会含义**：如果代码变成了一种像读写能力一样的通用技能，那么过去几十年围绕「程序员」这个身份建立起来的职业壁垒、薪资溢价、和组织权力结构，都将被重新洗牌。

这不是一夜之间发生的事。它是一个渐进的过程——模型在变强，工具在进化，组织在适应。但方向已经很清楚：**未来最重要的能力不是写代码，而是知道该写什么代码，以及怎么让代码被高效地写出来。**

Boris 用印刷术做的类比很到位。印刷术之后，抄写员这个职业消失了，但「作家」这个职业的价值反而上升了。不是因为作家能写出更好的字，而是因为他们有值得写的东西。

我们正站在软件行业的「古腾堡时刻」。代码正在被「印刷」，而真正稀缺的，永远是那些值得被印刷的内容。
