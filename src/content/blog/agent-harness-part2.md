## 前言

上篇我们聊了 Agent Harness 的编排层——执行循环、工具集成、记忆管理、多 Agent 协作。这些组件让一个 LLM 变成了能行动的 Agent。

但能跑起来只是第一步。更大的问题是：这个 Agent 到底做得好不好？它在不同任务上的表现如何？它会不会在某些情况下失控？它的成本能不能接受？

这篇文章是 Agent Harness 工程系列的下篇，聚焦两个主题：**评测体系**和**工程化实践**。前者回答「怎么衡量」，后者回答「怎么让它可靠」。读完你会对 Agent 系统的完整生命周期有一个清晰的认识。

## 1. 为什么需要专门的评测框架

### 1.1 传统 NLP 评测的局限

在 LLM 时代之前，NLP 任务的评测相对简单：给定输入，模型输出一个答案，和标准答案比对即可。

但对于 Agent，事情复杂得多：

**多步交互。** Agent 不是单次输出，而是经过多轮思考、行动、观察的循环。每一步都可能出错，也可能偶然走对。

**环境依赖。** Agent 的表现高度依赖运行环境——可用的工具、上下文窗口的长度、记忆的完整性。换一个环境，表现可能完全不同。

**开放结局。** 很多任务没有唯一正确答案。比如「帮我整理这个项目」，不同的整理方式都可能是合理的。

**长程可靠性。** Agent 可能在简单任务上表现很好，但在需要几十步的复杂任务中崩溃。短程评测无法暴露这个问题。

传统的准确率/F1 指标远远不够。我们需要一套专门针对 Agent 特性的评测框架。

### 1.2 评测的三个维度

一个好的 Agent 评测体系应该覆盖三个维度：

**能力（Capability）。** Agent 能不能完成任务？这是最基本的——通过率、成功率、任务完成度。

**效率（Efficiency）。** Agent 用了多少资源？包括 token 消耗、API 调用次数、运行时间、成本。

**鲁棒性（Robustness）。** Agent 在边界情况和异常输入下的表现如何？是否会陷入死循环？是否会产生有害输出？

这三个维度缺一不可。一个通过率高但成本极高的 Agent，和一个成本低但经常失控的 Agent，都不是好选择。

## 2. 主流评测基准

### 2.1 SWE-bench：软件工程能力的试金石

SWE-bench 由 Jimenez 等人在 2024 年提出，是目前评估 LLM 代码修复能力最权威的基准之一。

**设计思路。** 从真实 GitHub 仓库中收集 issue 和对应的 pull request（修复方案）。每个测试用例包含：

- 一个具体的 bug 描述（issue）
- 修复前的代码库状态
- 修复后的期望状态（通过 PR 中的测试验证）

Agent 的任务是：阅读 issue，理解代码库，生成修复补丁，使所有测试通过。

```text
┌─────────────┐     ┌──────────┐     ┌──────────────┐
│   Issue     │────▶│  Agent   │────▶│   Patch      │
│  (bug desc) │     │          │     │  (code diff) │
└─────────────┘     └──────────┘     └──────┬───────┘
                                            │
                                     ┌──────▼───────┐
                                     │  Run Tests   │
                                     │  Pass/Fail?  │
                                     └──────────────┘
```

**关键特点。**

- **真实世界复杂度。** 不是人工构造的 toy problem，而是真实项目中的真实 bug。
- **自动化验证。** 用单元测试判断修复是否正确，避免人工标注的主观性。
- **多语言支持。** 涵盖 Python、JavaScript、Java 等多种语言的仓库。

**局限性。** SWE-bench 主要评估单步代码修复能力，不涉及长期规划、多工具协作或跨文件重构。对于更复杂的软件工程任务，需要其他基准补充。

值得一提的是，SWE-bench 家族在 2024-2025 年间持续演化。**SWE-bench Lite** 是一个精选的 300 实例子集，用于更快速的评估。**SWE-bench Verified** 由 OpenAI 在 2024 年推出，包含 500 个人工验证的实例，质量更高。而 **SWE-bench Pro**（2025）进一步将任务扩展到数千个跨数十个活跃代码库的长程工程任务，要求 Agent 完成人类专家需要数天才能解决的复杂问题。

有趣的是，Xia 等人在 2024 年的 **Agentless** 工作中证明，一个简单的三步流程（定位→修复→打补丁）不需要复杂的 Agent 编排就能在 SWE-bench 上超越许多复杂的 Agent 方法。这提醒我们：不是所有问题都需要 Agent 化，有时候简单就是最好的。

> 📄 **Agentless: Demystifying LLM-based Software Engineering Agents**
> Xia, Deng, Dunn, Zhang. 2024.
> 证明简单的三步流程（定位、修复、打补丁）无需复杂 Agent 编排即可超越许多 Agent 方法。

> 📄 **SWE-bench: Can Language Models Resolve Real-World GitHub Issues?**
> Jimenez, Yang, Zhang, et al. NeurIPS 2024.
> 提出基于真实 GitHub issue 的代码修复评测基准，成为 LLM 软件工程能力的标准测试。

### 2.2 GAIA：通用 AI 助手的综合考试

GAIA（General AI Assistant Benchmark）由 Mialon 等人在 2023 年提出，目标是评估 Agent 在多样化现实任务上的综合能力。

**任务类型。** GAIA 包含三类任务：

- **信息检索与整合。** 例如「找出 2023 年诺贝尔物理学奖得主的出生地，并列出他们的主要贡献」。
- **数据操作与分析。** 例如「下载这个 CSV 文件，计算某列的平均值，并生成可视化图表」。
- **多步推理与规划。** 例如「比较三篇论文的方法差异，总结它们的共同点和分歧」。

**难度分级。** GAIA 将任务分为三个难度等级：

- **Level 1：** 单步或少量步骤，工具调用简单，答案明确。
- **Level 2：** 中等复杂度，需要多工具协作和一定的规划能力。
- **Level 3：** 高复杂度，需要长期规划、错误恢复和多轮迭代。

**评测指标。**

- **Pass@1：** 单次尝试的成功率。
- **Pass@k：** k 次尝试中至少一次成功的概率（反映重试能力）。
- **平均步数：** 完成任务所需的平均行动步数。
- **工具使用多样性：** 使用了多少种不同的工具。

GAIA 的价值在于它覆盖了 Agent 能力的多个侧面，而不是只关注某一类任务。这使得它成为评估通用 Agent 系统的重要基准。

> 📄 **GAIA: A Benchmark for General AI Assistants**
> Mialon, Dessì, Clochard, et al. 2023.
> 提出通用 AI 助手评测基准，涵盖信息检索、数据分析和多步推理任务。

### 2.3 WebArena 和 AgentBench：特定领域的深度评测

除了通用基准，还有一些针对特定领域的评测框架：

**WebArena。** 评估 Agent 在 Web 环境中的自主操作能力。任务包括在线购物、表单填写、账户管理等。Agent 需要通过浏览器自动化工具（如 Playwright）与网页交互。其多模态扩展 **VisualWebArena**（Koh et al., ACL 2024）进一步增加了 910 个需要视觉理解的任务，测试 Agent 能否同时处理图像和文本。

**AgentBench。** 由清华大学团队提出，覆盖操作系统、数据库、知识图谱等多个领域。每个子基准都有专门设计的任务和验证逻辑。

**tau-bench。** Yao 等人在 2024 年提出，评估 Agent 在真实客服对话场景中遵循工具使用策略和维护数据库状态的能力。即使是最好的模型，完成任务的比例也不到一半，凸显了 Agent 在策略遵从和状态管理方面仍然很弱。

**AgentBoard。** Ma 等人在 2024 年提出，引入了细粒度的进度指标（而不只是通过/失败），能更精确地诊断 Agent 在多轮交互中到底在哪一步失败。

**HotPotQA + ReAct。** 将多跳问答与 ReAct 框架结合，评估 Agent 在信息检索和推理链条上的表现。

这些专项基准的价值在于：它们能暴露通用基准看不到的弱点。一个在 GAIA 上表现很好的 Agent，可能在 WebArena 的表单填写任务中完全失败。

### 2.4 自建评测：针对你的场景

公开基准很重要，但它们不一定覆盖你的具体场景。工程实践中，通常需要自建评测集。

**设计原则。**

- **代表性。** 选取你实际业务中最常见的任务类型。
- **可自动化验证。** 尽量设计有明确正确答案的任务，或者可以用脚本验证的任务。
- **难度梯度。** 包含简单、中等、困难不同级别的任务，以便追踪进步。
- **回归测试。** 每次修改 Harness 后重新跑一遍，确保没有引入退化。

**示例：代码助手评测。**

如果你的 Agent 是一个代码助手，可以设计这样的评测集：

```python
test_cases = [
    {
        "task": "给这个函数添加类型注解",
        "input_code": "def add(a, b): return a + b",
        "expected": "def add(a: int, b: int) -> int: return a + b",
        "verify": "run_mypy"
    },
    {
        "task": "修复这个空指针异常",
        "input_code": "...",
        "expected_tests_pass": True,
        "verify": "run_unit_tests"
    }
]
```

自建评测的关键是：让它和你的实际使用场景紧密对齐。公开基准告诉你 Agent 的通用能力，自建评测告诉你它在你的业务中的表现。

## 3. 评测的工程实现

### 3.1 评测流水线

一个完整的评测流水线通常包含以下步骤：

```text
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 加载任务  │───▶│ 运行 Agent│───▶│ 收集结果  │───▶│ 验证答案  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
                                       │               │
                                       ▼               ▼
                                ┌──────────┐    ┌──────────┐
                                │ 记录日志  │    │ 计算指标  │
                                └──────────┘    └──────────┘
```

**关键点：**

- **隔离运行。** 每个任务应该在独立的环境中运行，避免相互干扰。
- **超时控制。** 设置合理的超时时间，防止某个任务卡住整个评测。
- **结果缓存。** 评测结果应该持久化，方便后续分析和对比。
- **并行执行。** 如果资源允许，可以并行运行多个任务，加速评测过程。

### 3.2 日志与追踪

评测不只是得到一个通过率数字。详细的日志和追踪对于调试和改进至关重要。

**应该记录的信息：**

- 每一轮的 thought、action、observation
- 工具调用的参数和返回值
- Token 消耗和 API 调用次数
- 错误信息和堆栈跟踪
- 最终答案和验证结果

**可视化工具。**

一些框架提供了内置的可视化工具：

- **LangSmith。** LangChain 官方的追踪平台，可以查看每条链的执行轨迹。
- **Phoenix。** Arize AI 提供的 LLM 观测平台，支持 trace 可视化和性能分析。
- **Weights & Biases。** 通用的 ML 实验追踪平台，也可以用于 Agent 评测。

这些工具让你能够「看到」Agent 的思考过程，而不仅仅是最终结果。这对于理解失败原因、优化提示词和调整架构非常有价值。

### 3.3 A/B 测试

当你改进了 Harness 的某个组件（比如换了更好的工具描述、调整了记忆检索策略），怎么知道改进真的有效？

答案是 A/B 测试：

1. 准备一个固定的评测集
2. 用旧版本跑一遍，记录结果
3. 用新版本跑一遍，记录结果
4. 对比两个版本的指标

关键是要保持评测集不变，否则无法区分是改进带来的提升还是任务难度变化导致的波动。

## 4. 成本控制：让 Agent 跑得便宜

### 4.1 成本构成

Agent 的成本主要来自三部分：

**LLM API 调用。** 这是大头。每一步 ReAct 循环都需要调用 LLM，复杂任务可能需要几十甚至上百步。

**工具调用成本。** 有些工具本身有成本，比如搜索 API、数据库查询、外部服务调用。

**基础设施成本。** 运行 Agent 的服务器、存储记忆的数据库、日志系统等。

其中 LLM API 是最难控制的，因为它和任务复杂度直接相关。一个简单的问答可能只需要几次调用，一个复杂的代码修复可能需要上百次。

### 4.2 降本策略

**缓存重用。** 如果同一个问题被问了多次，可以直接返回之前的答案，避免重复调用 LLM。这需要设计良好的缓存键（cache key），考虑输入、上下文和工具状态的组合。

**小模型兜底。** 对于简单任务，先用小模型（如 GPT-3.5-Turbo 或本地部署的小模型）尝试解决。只有当小模型失败或置信度低时，才调用大模型。这种分层策略可以显著降低成本。

**提前终止。** 设置合理的最大步数限制。如果 Agent 在一定步数内没有完成任务，就终止并返回部分结果或错误提示。这防止了无限循环带来的成本失控。

**批量处理。** 如果可能，把多个任务合并成一次 LLM 调用。比如同时处理多个相似的代码修复请求，让 LLM 一次性生成多个补丁。

**本地部署。** 对于高频使用的场景，考虑本地部署开源模型（如 LLaMA、Qwen）。虽然前期投入较大，但长期来看可能比持续调用 API 更便宜。

### 4.3 成本监控

无论采用什么降本策略，都需要实时监控成本。

**关键指标：**

- 每个任务的平均 token 消耗
- 每个任务的平均 API 调用次数
- 每个任务的平均运行时间
- 每日/每周总成本

**告警机制。**

设置成本阈值告警：当某个任务的成本超过预期，或者每日总成本异常升高时，及时通知团队介入调查。

## 5. 安全护栏：防止 Agent 失控

### 5.1 风险类型

Agent 相比普通 LLM 引入了新的安全风险：

**越权操作。** Agent 可能执行它不应该执行的操作，比如删除重要文件、发送未经授权的邮件、修改生产配置。

**敏感信息泄露。** Agent 可能在对话中无意中泄露了存储在记忆中的敏感信息，比如 API 密钥、用户隐私数据。

**注入攻击。** 恶意用户可能通过精心构造的输入，让 Agent 执行非预期操作。这类似于 SQL 注入，但是针对 Agent 的工具调用逻辑。

**资源滥用。** Agent 可能陷入死循环，不断调用昂贵的工具，导致成本飙升或服务不可用。

### 5.2 防护策略

**权限最小化。** Agent 应该只拥有完成其任务所需的最小权限。不要给它 root 访问权，不要让它能访问所有数据库表，不要让它能发送任意邮件。

**沙箱执行。** 对于代码执行类任务，使用沙箱环境（如 Docker 容器、gVisor、Firecracker microVM）。即使 Agent 生成了恶意代码，也只能在沙箱内执行，不会影响宿主系统。

**输入验证。** 对所有用户输入进行严格的验证和清洗。特别是那些会传递给工具调用的参数，需要做类型检查和范围限制。

**输出审核。** 在 Agent 的输出到达用户之前，经过一层审核过滤。可以用规则引擎（检查敏感关键词）或小模型（判断输出是否合适）来做这件事。

**速率限制。** 对 Agent 的 API 调用频率做限制，防止资源滥用。可以按用户、按会话或按全局设置不同的限流策略。

**审计日志。** 记录 Agent 的每一个操作，包括谁触发了它、它做了什么、结果如何。这对于事后追溯和安全分析至关重要。

### 5.3 人类介入

对于高风险操作，应该要求人类确认。

**审批流程。** 当 Agent 打算执行某个敏感操作（如删除数据、修改生产配置、发送大量邮件）时，暂停执行，等待人类审批。

**回滚机制。** 如果 Agent 的操作导致了意外后果，应该有快速回滚的手段。比如数据库操作的事务机制、文件操作的版本控制。

**紧急停止。** 提供一个「紧急停止」按钮，可以随时中断正在运行的 Agent。这在发现 Agent 行为异常时非常有用。

## 6. 生产级架构模式

### 6.1 微服务化

在生产环境中，Agent Harness 的各个组件通常被拆分为独立的微服务：

```text
┌──────────┐   ┌──────────┐   ┌──────────┐
│  API GW  │──▶│ Orchestr │──▶│ LLM Proxy│
└──────────┘   └──────────┘   └──────────┘
                    │              │
              ┌─────┴─────┐        │
              ▼           ▼        ▼
         ┌────────┐ ┌────────┐ ┌────────┐
         │ Tools  │ │ Memory │ │ Monitor│
         └────────┘ └────────┘ └────────┘
```

**好处：**

- 独立扩展。LLM 代理层和工具层可以分别扩容。
- 故障隔离。一个组件失败不会拖垮整个系统。
- 技术选型灵活。不同组件可以用不同的语言和框架实现。

### 6.2 事件驱动架构

对于需要异步处理的场景，事件驱动架构更合适：

**任务队列。** 用户请求放入队列（如 RabbitMQ、Kafka、SQS），Worker 从队列中取出任务执行。这样可以削峰填谷，应对突发流量。

**状态机。** Agent 的执行状态用状态机管理。每一步完成后发布事件，触发下一步。这样便于追踪和恢复。

**回调机制。** 对于长时间运行的任务，提供 webhook 回调接口。任务完成后主动通知调用方，而不是让调用方轮询。

### 6.3 可观测性

生产系统必须有完善的可观测性：

**指标（Metrics）。** 请求量、延迟、错误率、token 消耗、成本等关键指标实时采集和展示。

**日志（Logs）。** 结构化日志，包含请求 ID、用户 ID、任务 ID 等关键字段，便于追踪和排查。

**追踪（Traces）。** 分布式追踪，记录一个请求在整个系统中的流转路径。OpenTelemetry 是行业标准。

**告警（Alerts）。** 基于指标和日志设置告警规则，异常时及时通知值班人员。

## 7. 未来方向：Agent Harness 的演进

### 7.1 端到端训练

目前的 Agent Harness 大多是工程拼接——LLM 负责推理，外部代码负责工具调用和状态管理。未来可能会出现端到端训练的 Agent 模型，把工具调用和规划能力直接融入模型参数中。

DeepMind 的 Gato、Google 的 RT-2 等研究已经在探索这个方向。当模型本身具备了更强的规划和行动能力，Harness 的角色可能会从「编排者」变成「监督者」。

### 7.2 自我进化的 Harness

另一个有趣的方向是：让 Agent 自己改进它的 Harness。

比如，Agent 通过分析自己的失败案例，自动调整工具描述、优化记忆检索策略、甚至重新设计执行循环的逻辑。这类似于 AutoML 的思路，但是应用于 Agent 架构本身。

### 7.3 标准化协议

随着 Agent 生态的发展，可能会出现标准化的通信协议和工具接口规范。就像 HTTP 和 REST API 之于 Web 服务，Agent 之间也需要一套通用的「语言」来交流。

OpenAI 的 Function Calling、Anthropic 的 Tool Use 已经迈出了第一步。未来可能会有更统一的行业标准。

## 结语

Agent Harness 工程是一个快速演进的领域。过去两年，我们从简单的 ReAct 循环走到了多 Agent 协作、从手工评测走到了 SWE-bench 和 GAIA 这样的系统化基准、从 demo 级别的玩具走到了生产级的架构。

但挑战依然巨大：

- 评测仍然不够全面，很多重要能力难以量化
- 成本仍然高昂，复杂任务的经济可行性有待验证
- 安全仍然脆弱，新的攻击向量不断出现
- 可靠性仍然不足，Agent 在长程任务中的表现远未达到人类水平

这篇文章梳理了当前的最佳实践，但最好的实践永远是下一个。保持学习、持续实验、谨慎部署——这是在这个领域生存的基本法则。

希望这两篇文章能帮你建立起对 Agent Harness 工程的完整认知。如果你正在构建自己的 Agent 系统，欢迎分享你的经验和困惑。
