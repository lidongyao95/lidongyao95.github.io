## 前言

你让 AI 助手帮你调试一个前端组件。它试了十几种方法，花了 20 分钟，终于定位到问题——原来是一个 CSS 优先级冲突。

第二天，你又遇到了类似的问题。AI 助手完全不记得昨天的经验，又从第一性原理开始推理，又花了 20 分钟。

这就是当前 AI Agent 的一个根本问题：**每次对话都是一张白纸。** 模型不会从过去的成功中学习，不会把踩过的坑记下来，不会把解决过的问题提炼成可复用的方案。

Skill（技能）就是为了解决这个问题而设计的——它是一种把**程序性知识**封装成可复用单元的方法，让 Agent 能够积累和调用经验。

## 1. 从工具到技能：一次认知升级

### 1.1 工具是什么

在上一篇 MCP 的博客中，我们讨论了工具（Tools）——模型可以调用的外部能力。搜索网页、读取文件、执行代码，这些都是工具。

工具的特点是：**原子性**。一次调用完成一个具体动作，不管上下文，不管业务逻辑。

```text
工具 = 原子操作
```

### 1.2 技能是什么

技能（Skill）是比工具高一层的抽象。它不是「做一次动作」，而是「完成一个任务」。

一个技能可能包含：

- 多次工具调用
- 条件判断和分支逻辑
- 错误处理和重试策略
- 质量检查和验证步骤
- 领域特定的最佳实践

```text
技能 = 任务流程 + 领域知识 + 最佳实践
```

### 1.3 一个类比：菜谱与食材

工具和技能的关系，就像食材和菜谱的关系。

食材（工具）是基础单元——你可以单独使用面粉、鸡蛋、糖。但如果你不知道比例、顺序和火候，做出来的东西可能很难吃。

菜谱（技能）是经验的封装——「做戚风蛋糕：先打发蛋白，蛋黄糊用低筋面粉，烤箱预热 170°C，烤 35 分钟」。这是一套经过验证的流程，包含了「为什么」和「怎么做」的知识。

Agent 如果只有工具，就像一个有所有食材但不会做饭的人。有了技能，它就能按照菜谱做出成品。

## 2. 程序性知识 vs 陈述性知识

### 2.1 两种知识类型

认知科学把知识分为两类：

**陈述性知识（Declarative Knowledge）。** 关于「是什么」的知识。比如「Transformer 的隐藏维度通常是 4096」「Python 的 GIL 限制了多线程性能」。

**程序性知识（Procedural Knowledge）。** 关于「怎么做」的知识。比如「如何调试一个内存泄漏」「如何部署一个 React 应用到生产环境」。

LLM 天然擅长陈述性知识——它在预训练中阅读了海量文本，记住了大量事实。但程序性知识很难从文本中学到，因为它通常存在于**实践中**，而不是文档里。

### 2.2 为什么程序性知识难获取

一个资深工程师的调试方法，很少写在官方文档里。它散落在：

- 团队内部的 wiki 和 runbook
- 多年的踩坑经验
- 口口相传的「秘诀」
- Stack Overflow 的零散回答

这些知识很难通过预训练学到，因为：

**缺乏显式表达。** 经验丰富的工程师不会写「当我遇到 X 时，我通常先检查 Y，再看 Z，如果不行就试试 W」——这种推理过程是隐性的。

**高度上下文依赖。** 同一个问题在不同项目中可能有完全不同的解决路径。

**快速变化。** 工具和框架在持续更新，今天的最佳实践明天可能就过时了。

Skill 的设计目标就是让 Agent 能够**积累和复用**程序性知识——把隐性的经验显性化，把一次性的解决方案变成可重复的技能。

## 3. Skill 的解剖

### 3.1 一个 Skill 的结构

一个典型的 Skill 包含以下要素：

```text
Skill: debug-memory-leak
├── name: "内存泄漏调试"
├── description: "当应用出现内存持续增长时使用的系统化调试流程"
├── version: 1.0.0
│
├── 触发条件 (triggers)
│   ├── "内存使用率持续上升"
│   ├── "OOM killed"
│   └── "heap out of memory"
│
├── 执行步骤 (steps)
│   ├── 1. 确认现象：检查监控数据，确认内存确实在增长
│   ├── 2. 定位范围：通过 heap snapshot 对比，找到增长的对象
│   ├── 3. 追踪引用：分析对象引用链，找到泄漏源头
│   ├── 4. 验证修复：修复后运行一段时间，确认内存稳定
│   └── 5. 记录教训：更新 runbook，避免再次踩坑
│
├── 工具依赖 (tools)
│   ├── profiler（性能分析器）
│   ├── monitoring（监控面板）
│   └── logging（日志系统）
│
└── 常见陷阱 (pitfalls)
    ├── "不要只看总内存，要看 heap breakdown"
    ├── "GC 后的内存才是真实内存"
    └── "某些框架的内存池会持续增长，不一定是泄漏"
```

### 3.2 关键设计原则

**可触发。** Skill 需要能被自动识别和调用。这要求描述足够清晰，触发条件足够明确。不能写「当你遇到问题时使用」，而要写「当看到 X 错误码或 Y 症状时使用」。

**可组合。** 复杂的任务可以通过组合多个 Skill 来完成。比如「部署到生产环境」可以拆分为「运行测试」→「构建镜像」→「推送镜像」→「滚动更新」四个子 Skill。

**可演进。** Skill 不是一成不变的。随着实践经验的积累，应该持续更新——添加新的步骤、修正过时的建议、记录新的陷阱。

**可验证。** 每个 Skill 应该有明确的成功标准。「调试完成」不够好，「内存使用率在修复后 24 小时内保持稳定」才是可验证的。

## 4. Skill 的三种来源

### 4.1 人工编写

最直接的方式是让人类专家把经验写成 Skill。

**优点：** 质量可控，可以包含深度洞察和微妙判断。

**缺点：** 成本高，扩展性差。一个团队的经验很难覆盖所有场景。

**适用场景：** 核心业务流程、高风险操作、安全关键任务。这些场景下，错误成本远高于编写成本。

NVIDIA 的 **Voyager**（Wang et al., TMLR 2024）是这个方向的先驱——它在 Minecraft 中让 Agent 自主探索，每学会一项新技能就将其编码为可验证的代码，存入持续增长的技能库。这些技能可以组合使用，而且不会灾难性遗忘。Voyager 证明了：技能库的核心价值在于**可组合性**和**持续积累**。

### 4.2 从对话中提炼

当 Agent 成功解决了一个复杂问题时，可以把解决过程自动提炼为 Skill。

```text
对话记录:
用户: "我的 Next.js 应用部署后 CSS 样式丢失了"
Agent: [尝试 5 种方法]
Agent: "发现问题了！Next.js 13+ 的 App Router 需要在 layout.tsx 
        中显式 import globals.css，而不是在 _app.tsx 中"

→ 自动提炼 Skill:
  name: "nextjs-css-missing"
  description: "Next.js 13+ App Router 部署后 CSS 样式丢失的排查"
  steps: ["检查 layout.tsx 是否 import globals.css",
          "确认 CSS 文件路径正确", ...]
```

**优点：** 可以从真实的成功案例中学习，覆盖面广。

**缺点：** 自动提炼的质量不一定高，可能丢失关键上下文或提取出错误的泛化。

**适用场景：** 高频重复问题、社区常见 bug、标准工作流程。

### 4.3 从文档和代码中生成

通过分析项目文档、代码仓库、配置文件，自动生成 Skill。

比如，分析一个项目的 CI/CD 配置，可以生成：

```text
Skill: deploy-to-production
description: "本项目的生产部署流程"
steps: ["运行 npm test", "构建 Docker 镜像", "推送到 registry",
        "执行 kubectl apply", "等待 rollout 完成", "运行 smoke test"]
tools: [docker, kubectl, npm]
```

**优点：** 与项目强绑定，准确度高。

**缺点：** 只能提取显式记录的流程，无法捕捉隐性知识。

**适用场景：** 项目特定的工作流、团队协作规范、基础设施操作。

## 5. Skill 的检索与调用

### 5.1 触发时机

Skill 不是一直开着的。Agent 需要在合适的时机决定使用哪个 Skill。

**显式调用。** 用户直接说「用 debug-memory-leak 技能帮我排查」。最简单，但要求用户知道技能的存在。

**语义匹配。** 用户描述问题，Agent 自动匹配最相关的 Skill。这通常通过向量相似度实现——把用户问题和所有 Skill 的 description/triggers 编码成向量，找最接近的。

**多阶段检索。** 先用粗粒度的分类过滤（比如「这是调试类技能还是开发类技能」），再用细粒度的匹配找到具体 Skill。这比全量搜索更高效。

### 5.2 上下文注入

找到 Skill 后，需要把 Skill 的内容注入到 Agent 的上下文中。

**完整注入。** 把 Skill 的所有内容（描述、步骤、工具、陷阱）都放进 prompt。简单直接，但可能超出上下文窗口。

**按需注入。** 只注入 Skill 的概要，当 Agent 执行到具体步骤时再加载详细内容。更节省 token，但需要额外的检索逻辑。

**分层注入。** 对于组合 Skill，先注入顶层编排逻辑，子 Skill 按需加载。适合复杂任务。

### 5.3 执行与监控

Skill 执行过程中需要监控进度和质量：

**步骤追踪。** 记录每个步骤的开始时间、结束时间、执行结果。

**失败处理。** 某个步骤失败时，是重试、跳过、还是回滚？Skill 应该定义清晰的错误处理策略。

**质量门控。** 关键步骤完成后进行质量检查，不通过则不进入下一步。比如「代码生成」步骤后必须有「代码审查」步骤。

## 6. Skill 的组合与编排

### 6.1 串行组合

最简单的组合方式是按顺序执行多个 Skill：

```text
用户需求: "给这个项目加上单元测试并部署"

→ Skill A: add-unit-tests
  ├── 分析代码结构
  ├── 生成测试用例
  └── 运行测试确认通过

→ Skill B: deploy-to-staging
  ├── 构建应用
  ├── 部署到 staging 环境
  └── 运行集成测试

→ Skill C: promote-to-production
  ├── 确认 staging 测试通过
  ├── 执行生产部署
  └── 运行 smoke test
```

### 6.2 并行组合

当多个 Skill 之间没有依赖关系时，可以并行执行：

```text
用户需求: "全面检查这个代码仓库"

┌── Skill A: code-quality-check ──┐
│   (代码质量检查)                │
├── Skill B: security-scan ───────┤──▶ 汇总报告
│   (安全扫描)                    │
├── Skill C: performance-audit ───┤
│   (性能审计)                    │
└── Skill D: dependency-check ────┘
    (依赖检查)
```

### 6.3 条件分支

根据中间结果选择不同的后续 Skill：

```text
用户需求: "帮我调试这个报错"

→ Skill: diagnose-error
  ├── 分析错误日志
  ├── 定位可能原因
  │
  ├── if 原因是配置问题:
  │   └── Skill: fix-config
  │
  ├── if 原因是代码 bug:
  │   └── Skill: debug-code
  │
  └── if 原因是环境问题:
      └── Skill: fix-environment
```

## 7. Skill 的演进与维护

### 7.1 版本管理

Skill 应该像代码一样有版本管理：

```text
Skill: deploy-kubernetes
version: 2.1.0
changelog:
  - 2.1.0: 添加蓝绿部署支持
  - 2.0.0: 迁移到 Kubernetes 1.28，废弃旧版 API
  - 1.0.0: 初始版本，基础滚动更新
```

当底层工具或最佳实践发生变化时，更新 Skill 版本。旧版本可以保留作为回退选项。

### 7.2 效果评估

定期评估 Skill 的使用效果：

**成功率。** 使用这个 Skill 的任务有多少成功完成？

**效率。** 平均完成时间是多少？相比不使用 Skill 时有多少提升？

**用户反馈。** 用户对 Skill 的输出满意吗？哪些步骤经常被修改或跳过？

**退化检测。** 随着环境变化，Skill 的效果可能在下降。需要监控成功率趋势，及时发现问题。

### 7.3 持续改进

Skill 不是一次性交付，而是持续演进的知识资产：

**从失败中学习。** 当 Skill 执行失败时，分析原因，更新 Skill 以覆盖这类场景。

**从成功中提炼。** 当 Skill 特别成功地解决了某个问题，记录成功因素，推广到类似场景。

**社区贡献。** 对于通用 Skill，可以在社区中分享和讨论，收集反馈，持续改进。但 Jiang 等人（2026）在首篇 Agent Skills 的系统化知识论文（SoK）中指出，技能市场也面临安全风险——恶意技能可能伪装成合法工具，窃取用户凭证或注入恶意指令。技能的分发和审核机制是未来需要重点解决的问题。

## 8. Skill 与 Agent Harness 的关系

### 8.1 在架构中的位置

回顾上篇 Agent Harness 的架构图，Skill 处于记忆管理和执行循环之间：

```text
┌─────────────────────────────────────┐
│            Agent Harness            │
│                                     │
│  ┌─────────┐  ┌─────────────────┐  │
│  │ 执行循环 │◀▶│   Skill 引擎    │  │
│  │         │  │  (检索/注入/执行)│  │
│  └────┬────┘  └────────┬────────┘  │
│       │                │            │
│  ┌────▼────┐     ┌─────▼──────┐    │
│  │ 工具层  │     │ Skill 存储  │    │
│  │ (MCP)  │     │ (向量数据库 │    │
│  └─────────┘     │  或文件系统)│    │
│                  └────────────┘    │
└─────────────────────────────────────┘
```

### 8.2 与工具的关系

Skill 和 Tool 是互补的：

- **Tool** 提供原子能力（读文件、调 API）
- **Skill** 编排这些能力完成具体任务（调试、部署、分析）

一个 Skill 可能使用多个 Tool，但一个 Tool 调用不构成一个完整的 Skill。

### 8.3 与记忆的关系

Skill 是一种**结构化的长期记忆**。相比通用的向量记忆（存储过去的对话片段），Skill 是经过提炼和组织的高阶知识。

```text
记忆：上次用户遇到 X 问题时，我们通过 Y 方法解决了
Skill：当遇到 X 类问题时，标准流程是 1→2→3→4，常见陷阱是 A/B/C
```

记忆是经验的原始记录，Skill 是经验的提炼总结。两者结合使用效果最好——Skill 提供标准流程，记忆提供个性化上下文。

## 结语

Skill 的核心洞察是：AI Agent 的价值不只在于单次推理的能力，更在于**积累和复用经验的能力**。

一个没有 Skill 的 Agent，每次对话都从零开始。它可能在单次任务中表现很好，但无法从过去的成功和失败中学习。就像一个每天失忆的天才——能力很强，但永远在重复犯同样的错误。

Skill 让 Agent 有了「记忆」和「成长」的可能。它把隐性的程序性知识显性化，把一次性的解决方案变成可复用的标准流程，把个人的经验变成团队的知识资产。

这听起来很简单，但实现起来充满挑战：如何保证 Skill 的质量？如何处理 Skill 之间的冲突？如何让 Skill 跟上快速变化的技术环境？这些问题没有银弹，需要持续的工程实践和社区协作。

但方向是对的。当 Agent 真正能够积累和调用经验时，它就不再只是一个工具，而是一个能够成长的伙伴。
