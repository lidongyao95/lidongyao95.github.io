## 想象这样一个场景

你在公司里用 AI 助手查制度：

> 员工出差住宿每天最多能报多少钱？

AI 助手自信地回答：

> 根据公司制度，出差住宿报销上限为 500 元/天。

听起来很合理。但你去翻了制度文件，发现：

> 上海属于一类城市，住宿上限 **600 元/天**。

AI 说的 500 是哪来的？它编的。

这不是 bug，这是大模型的工作方式：它靠训练数据里的记忆回答问题，像个**闭卷考试的学生**。考试时没带课本，只能凭记忆写答案——记对了就得分，记错了就编一个看似合理的答案。

大模型有三个天然限制：

- 它不知道训练后才出现的新信息
- 它很难完整记住企业内部的私有知识
- 它生成答案时不一定能给出可追溯来源

所以当你问「公司最新的报销制度是什么」「这份合同里有哪些风险条款」，模型很可能凭记忆回答，也可能编造一个看起来很专业的答案——而你无从验证。

**RAG（Retrieval-Augmented Generation，检索增强生成）**就是来解决这个问题的。它的核心思路朴素到一句话就说完了：

> 不要只依赖模型参数里的记忆。先从外部知识库检索相关资料，再把资料交给模型生成答案。

也就是说，让模型从闭卷变成开卷。

## 传统问答 vs RAG 问答

传统的大模型问答流程很简单：

```text
用户问题 → 大模型 → 答案
```

所有知识都来自模型参数。模型能不能答对，取决于它训练时是否见过相关内容。

RAG 在中间插了一步：

```text
用户问题 → 检索系统 → 相关文档片段 → 大模型 → 带依据的答案
```

模型不再孤立地回答问题。它像一个会查资料的助手——先拿到相关资料，再基于资料组织语言。

## RAG 的两条链路

一个 RAG 系统可以拆成两条链路：

```text
离线链路：文档 → 清洗 → 切分 → 向量化 → 写入索引
在线链路：问题 → 检索 → 重排 → 组装上下文 → 生成答案
```

离线链路处理知识库构建，在线链路处理每次提问。两条链路通过向量数据库连接——离线链路把知识存进去，在线链路从里面找出来。

## 检索的直觉：意思相近但字面不一样

RAG 之所以有效，很大程度是因为检索能解决一个关键词搜索很难搞的问题：**意思相近但字面不一样**。

比如你问：

> 新员工入职后多久可以申请年假？

制度里写的可能是：

> 员工连续工作满一年后享有带薪年休假。

这两句话没有一个完全相同的关键词，但语义相关。关键词搜索可能找不到，而向量检索通过 embedding 能把语义相近的文本在向量空间里放在一起——这样就能召回。

不过向量检索也有短板。如果你问的是一个错误码 `ERR_PAYMENT_402`、一个 API 名称、一个法条编号——这些需要精确匹配，向量检索可能不如关键词检索稳定。

所以工程上常用**混合检索**：向量检索负责语义匹配，关键词检索负责精确匹配，两者合并后再交给排序模块。这样既不漏掉语义相近的结果，也不丢精确命中的结果。

## 一个完整例子

假设我们要做一个公司制度问答助手。用户问：

> 去上海出差，住宿费每天最多能报多少？

### 第一步：检索

系统从差旅制度、报销说明、城市等级表中召回相关片段。

### 第二步：重排

初步检索可能召回几十个片段，但不一定按相关性排序。Reranker 重新判断每个片段和问题的相关程度，发现「城市等级表」和「住宿标准」最相关，把它们排到前面。

### 第三步：组装上下文

把最相关的片段整理好，交给模型：

```text
[1] 城市等级表：上海属于一类城市。
[2] 住宿标准：一类城市住宿费上限为 600 元/天。
```

### 第四步：生成答案

```text
根据现有制度，上海属于一类城市，住宿费报销上限为 600 元/天。依据：[1][2]
```

这个答案不是模型凭记忆说出来的，而是由检索结果支撑的。模型的角色从「凭记忆回答」变成了「基于资料回答」。

## RAG vs 微调：查资料 vs 学规矩

RAG 和微调经常被放在一起讨论，但它们做的事完全不同。

一个直觉类比：

- **RAG 让模型查资料**——开卷答题，答案有依据
- **微调让模型学规矩**——改行为方式，不是改知识来源

| 维度 | RAG | 微调 |
|------|-----|------|
| 主要目标 | 引入外部知识 | 改变模型行为 |
| 知识更新 | 更新知识库即可 | 通常需要重新训练 |
| 可追溯性 | 可以引用来源 | 不天然可追溯 |
| 适合场景 | 知识库问答、文档检索 | 输出格式、语气风格、任务规范 |

在真实系统中，它们经常一起用。比如一个客服助手可以通过微调学会固定话术和服务边界，同时通过 RAG 查询最新产品手册和售后政策。

## 结语

RAG 的本质就一句话：**让模型从闭卷答题变成开卷答题**。

它让模型不必把所有知识都记在参数里，而是可以在回答前查找外部资料。这样既能使用最新和私有知识，也能提供引用来源，降低幻觉风险。

但 RAG 不是银弹——demo 能跑不代表生产能用。检索没找对资料，再强的模型也只能基于错误上下文回答。生成质量的上限，往往由检索质量决定。
