RAG 技术详解：让大模型用上外部知识

前言

大语言模型很强，但它有三个天然限制：

  • 它不知道训练后才出现的新信息。
  • 它很难完整记住企业内部的私有知识。
  • 它生成答案时不一定能给出可追溯来源。

如果我们直接问模型「公司最新的报销制度是什么」「某个项目的接口文档怎么写」「这份合同里有哪些风险」，模型可能会凭记忆回答，也可能会编造一个看似合理的答案。

**RAG（Retrieval-Augmented Generation，检索增强生成）**就是为了解决这个问题而出现的技术路线。它的核心思路很朴素：

不要只依赖模型参数里的记忆。
先从外部知识库检索相关资料，再把资料交给模型生成答案。

这篇文章会从底层原理到工程实现，完整讲清楚 RAG 系统到底怎么工作、为什么有效、哪里容易翻车，以及如何评估它是否真的可靠。

1. RAG 解决的是什么问题

传统大模型问答大致是这样：

用户问题 → 大模型 → 答案

这里的所有知识都来自模型参数。模型能不能答对，取决于它训练时是否见过相关内容，以及这些内容是否被稳定地编码进参数。

RAG 把流程改成：

用户问题 → 检索系统 → 相关文档片段 → 大模型 → 带依据的答案

也就是说，模型不再孤立地回答问题，而是像一个会查资料的助手。它先拿到相关资料，再基于资料组织语言。

RAG 适合解决的问题包括：

  • 企业知识库问答
  • 文档助手和合同审阅
  • 技术文档搜索与解释
  • 客服知识库问答
  • 研究资料整理
  • 法规、制度、流程类问答
  • 需要引用来源的专业回答

RAG 不适合把模型变成数据库。它更像是在「搜索」和「写作」之间搭桥：检索负责找资料，生成负责理解和表达。

2. RAG 的基本架构

一个典型 RAG 系统可以拆成两条链路：

离线链路：文档 → 清洗 → 切分 → 向量化 → 写入索引
在线链路：问题 → 检索 → 重排 → 组装上下文 → 生成答案

2.1 离线链路：把知识变成可检索资产

离线链路处理的是知识库构建。

比如我们有一批 PDF、Word 文档、网页、Markdown、飞书文档和数据库记录。系统要先把它们转成统一的文本结构，然后切成适合检索的小片段。

流程大致如下：

原始文档
  ↓
文本抽取与清洗
  ↓
按章节、段落或语义切分
  ↓
生成 embedding 向量
  ↓
写入向量数据库和元数据索引

这里最关键的产物不是完整文档，而是一个个带有元数据的 chunk。

一个 chunk 通常长这样：

{
  "id": "policy-2026-001#section-3",
  "text": "员工差旅住宿标准按照城市等级划分...",
  "source": "2026 差旅报销制度.pdf",
  "section": "3. 住宿标准",
  "updated_at": "2026-05-12",
  "embedding": [0.012, -0.081, 0.034]
}

文本负责被模型阅读，embedding 负责被检索系统搜索，元数据负责过滤、引用和权限控制。

2.2 在线链路：从问题到答案

用户提问时，系统不会马上把问题丢给大模型，而是先检索。

用户问题
  ↓
问题改写或扩展
  ↓
向量检索 / 关键词检索 / 混合检索
  ↓
候选片段重排
  ↓
上下文组装
  ↓
大模型生成答案
  ↓
返回答案和引用来源

一个工程上更完整的 RAG 系统通常会包含：

  • Query Understanding：理解用户真正想问什么。
  • Retriever：召回可能相关的资料。
  • Reranker：重新排序候选资料。
  • Context Builder：把资料压缩成模型能读的上下文。
  • Generator：基于上下文生成答案。
  • Evaluator：评估答案是否相关、忠实、完整。

3. Embedding：把语义变成向量

RAG 的关键技术之一是 embedding。

Embedding 模型会把一段文本转成一个高维向量。语义相近的文本，向量距离也更近。

例如：

"如何申请差旅报销"
"出差费用怎么报销"
"差旅费用审批流程是什么"

这三句话表面用词不同，但语义接近。好的 embedding 模型会把它们放在向量空间中相近的位置。

检索时，系统会把用户问题也转成向量，然后在向量数据库里找距离最近的 chunk。

常见的相似度计算包括：

  • 余弦相似度
  • 内积
  • 欧氏距离

从直觉上看，向量检索解决的是「意思相近但字面不一样」的问题。这是传统关键词搜索比较吃力的地方。

4. 文档切分：RAG 成败的第一道关

很多 RAG 系统效果差，不是因为模型不行，而是文档切分做得粗糙。

切分太大，会带来两个问题：

  • 检索命中的片段里混入太多无关内容。
  • 上下文窗口被浪费，模型读到的有效信息变少。

切分太小，也会有问题：

  • 单个片段缺少上下文，模型无法理解完整含义。
  • 一个答案需要多个片段拼起来，检索难度变高。

4.1 常见切分策略

固定长度切分

按照 token 数或字符数切分，比如每 500 个 token 一段。这种方式简单稳定，但容易切断语义。

按结构切分

根据标题、章节、段落、表格和列表切分。技术文档、制度文件、论文和合同更适合这种方式。

语义切分

通过句子相似度或主题变化判断边界，尽量让每个 chunk 表达一个完整意思。

滑动窗口切分

相邻 chunk 保留一部分重叠内容，避免关键信息刚好落在边界处。

Chunk 1: A B C D E
Chunk 2: D E F G H
Chunk 3: G H I J K

重叠可以提升召回率，但也会增加索引体积和重复信息。

4.2 推荐原则

切分时可以遵循几个原则：

  • 优先保留文档的自然结构。
  • chunk 内部表达一个相对完整的语义单元。
  • 保留标题、来源、页码、章节等元数据。
  • 对表格、代码、公式和图片说明做单独处理。
  • 用真实问题测试检索效果，而不是只看切分是否整齐。

RAG 不是把文档切碎这么简单，而是把知识组织成适合机器检索和模型阅读的形态。

5. 检索：找得到比生成更重要

如果检索阶段没有找对资料，后面的生成模型再强也只能基于错误上下文回答。

这就是 RAG 的一个基本原则：

生成质量的上限，往往由检索质量决定。

5.1 向量检索

向量检索适合语义匹配。

比如用户问：

新员工入职后多久可以申请年假？

文档里可能写的是：

员工连续工作满一年后享有带薪年休假。

这两个句子没有完全相同的关键词，但语义相关。向量检索更容易召回。

5.2 关键词检索

关键词检索适合精确匹配。

例如：

  • 产品型号
  • API 名称
  • 法条编号
  • 人名
  • 错误码
  • 专有名词

如果用户问 ERR_PAYMENT_402，向量检索可能不如关键词检索稳定。

5.3 混合检索

工程上常用的是混合检索：

向量检索召回语义相关结果
关键词检索召回精确匹配结果
合并候选集合
再交给 reranker 排序

这种方式兼顾语义理解和精确命中，是企业知识库 RAG 中非常常见的方案。

6. Rerank：从「可能相关」到「真正相关」

初步检索通常会召回几十到几百个候选 chunk。它们只是「可能相关」，顺序未必可靠。

Reranker 的任务是重新判断：

用户问题 和 文档片段 到底有多相关？

Embedding 检索一般是把问题和文档分别编码成向量，再计算距离。Reranker 通常会同时读取问题和文档片段，判断两者关系，因此更精细，但成本也更高。

一个常见流程是：

向量检索 top 50
关键词检索 top 50
合并去重
reranker 排序
保留 top 5 到 top 10

Rerank 的价值在于减少「看起来相似但回答不了问题」的片段进入上下文。

7. 上下文组装：不是把检索结果直接塞进去

很多 RAG 系统的坏味道是：检索到什么就全部塞给模型。

这样会带来几个问题：

  • 上下文太长，浪费 token。
  • 多个片段互相冲突，模型不知道信谁。
  • 片段顺序混乱，模型理解成本变高。
  • 引用来源不清楚，答案不可追溯。

好的上下文组装应该考虑：

  • 按相关性排序
  • 保留文档标题和章节
  • 合并相邻片段
  • 去掉重复片段
  • 对长文档做摘要压缩
  • 对不同来源标明编号

一个常见 prompt 结构如下：

你是一个严谨的知识库问答助手。
只能依据给定资料回答。
如果资料不足，请说明无法确定。

用户问题：
{question}

参考资料：
[1] 来源：差旅制度.pdf / 3. 住宿标准
{chunk_1}

[2] 来源：报销流程.md / 2. 审批节点
{chunk_2}

请给出答案，并标注引用来源编号。

这里的重点是让模型知道：它不是自由发挥，而是在做基于证据的回答。

8. 生成：让模型忠实于资料

RAG 中的大模型主要负责三件事：

  • 理解用户问题
  • 阅读检索到的资料
  • 组织答案

它不应该凭空补充没有依据的内容。

为了降低幻觉，可以在 prompt 里加入约束：

如果参考资料没有直接支持答案，请回答「根据现有资料无法确定」。
不要使用参考资料之外的信息。
涉及数字、日期、流程、责任人时必须引用来源。

但只靠 prompt 不够。还需要在系统层面做防护：

  • 检索结果为空时不要强行回答。
  • 低置信度时提示人工确认。
  • 关键场景保留引用和原文片段。
  • 对答案做事实一致性检查。

RAG 的目标不是让模型永远自信，而是让它在资料不足时学会诚实。

9. RAG 与微调的区别

RAG 和微调经常被放在一起讨论，但它们解决的问题不同。

维度RAG微调
主要目标引入外部知识改变模型行为
知识更新更新知识库即可通常需要重新训练或适配
可追溯性可以引用来源不天然可追溯
适合场景知识库问答、文档检索、实时资料输出格式、语气风格、任务规范
风险检索不到或检索错过拟合、遗忘、成本高

简单说：

  • RAG 让模型查资料。
  • 微调让模型学规矩。

在真实系统中，它们经常一起使用。比如一个客服助手可以通过微调学会固定话术和服务边界，同时通过 RAG 查询最新产品手册和售后政策。

10. 常见失败模式

RAG 系统失败时，通常可以从以下几个环节定位。

10.1 文档没有被正确解析

PDF 表格错乱、扫描件 OCR 错误、标题层级丢失，都会影响后续检索。

如果文档解析阶段已经把内容弄坏了，后面再好的 embedding 和模型也救不回来。

10.2 chunk 切分不合理

问题需要的关键信息被拆到多个 chunk 中，或者 chunk 里混入太多无关内容，都会导致答案不稳定。

10.3 检索召回不足

系统没有找到真正相关的资料。表现为模型回答泛泛而谈，或者说「资料不足」，但其实知识库里有答案。

10.4 重排不准确

相关资料被召回了，但排在后面，没有进入最终上下文。

10.5 上下文冲突

知识库里有旧版本和新版本文档，模型同时读到两个冲突片段，于是给出混合答案。

这时需要通过元数据过滤、版本控制和时间优先级解决。

10.6 生成阶段不忠实

模型读到了正确资料，但回答时加入了没有依据的推断。

这需要 prompt 约束、引用检查和答案验证共同处理。

11. 如何评估 RAG 系统

评估 RAG 不能只看最终答案，还要拆开看每一层。

11.1 检索评估

检索层关注：正确资料有没有被找出来？

常见指标包括：

  • Recall@K：正确片段是否出现在前 K 个结果中。
  • Precision@K：前 K 个结果里有多少是真的相关。
  • MRR：第一个正确结果排在多前面。

如果 Recall@10 很低，说明召回有问题；如果 Recall@50 高但最终答案差，可能是 rerank 或上下文组装有问题。

11.2 生成评估

生成层关注：答案是否好用？

可以从几个维度评估：

  • 相关性：是否回答了用户问题。
  • 忠实性：是否完全依据检索资料。
  • 完整性：是否遗漏关键条件。
  • 可读性：表达是否清楚。
  • 引用准确性：引用编号是否真的支持对应结论。

11.3 端到端评估

最终还是要用真实问题集测试。

一个实用的评估集应该包含：

  • 高频常见问题
  • 长尾问题
  • 多跳问题
  • 需要精确数字的问题
  • 资料不足的问题
  • 多版本冲突问题
  • 容易被关键词误导的问题

RAG 系统上线后，还应该持续记录：

  • 用户问题
  • 检索到的 chunk
  • 最终答案
  • 用户反馈
  • 人工修正结果

这些数据会反过来帮助优化切分、检索、重排和 prompt。

12. 工程落地架构

一个生产级 RAG 系统通常不只是一个「向量数据库 + LLM」。

更完整的架构可能是：

文档接入层
  ├── 文件上传
  ├── 网页抓取
  ├── 企业文档同步
  └── 数据库同步

处理层
  ├── 文本抽取
  ├── 清洗去噪
  ├── 结构解析
  ├── chunk 切分
  └── embedding 生成

索引层
  ├── 向量索引
  ├── 关键词索引
  ├── 元数据索引
  └── 权限索引

查询层
  ├── query 改写
  ├── 混合检索
  ├── rerank
  ├── 上下文压缩
  └── 答案生成

治理层
  ├── 权限控制
  ├── 引用追踪
  ├── 质量评估
  ├── 日志审计
  └── 反馈闭环

企业场景里，权限控制尤其重要。用户只能检索自己有权限看的文档，否则 RAG 会变成信息泄露通道。

13. 一个完整例子

假设我们要做一个公司制度问答助手。

用户问：

去上海出差，住宿费每天最多能报多少？

系统流程如下：

第一步：问题理解

识别关键词和约束：

  • 城市：上海
  • 场景：出差
  • 费用类型：住宿费
  • 问题意图：查询报销上限

第二步：检索

系统从差旅制度、报销说明、城市等级表中召回相关片段。

第三步：重排

Reranker 发现「城市等级表」和「住宿标准」最相关，把它们排到前面。

第四步：上下文组装

[1] 城市等级表：上海属于一类城市。
[2] 住宿标准：一类城市住宿费上限为 600 元/天。

第五步：生成答案

根据现有制度，上海属于一类城市，住宿费报销上限为 600 元/天。依据：[1][2]

这个答案不是模型凭记忆说出来的，而是由检索结果支撑的。

14. 实践建议

如果要从零搭建 RAG 系统，可以按这个顺序推进：

  1. 先做 50 到 100 个真实问题，建立评估集。
  2. 做文档解析和结构化清洗，不要急着接大模型。
  3. 尝试几种 chunk 策略，用评估集看召回率。
  4. 同时使用向量检索和关键词检索。
  5. 加入 reranker，观察 top K 结果质量变化。
  6. 设计带引用的上下文模板。
  7. 让模型在资料不足时明确拒答。
  8. 上线后记录检索结果和用户反馈。
  9. 定期清理旧版本和重复文档。
  10. 对高风险答案保留人工审核。

RAG 的难点不在 demo，而在持续维护知识质量、检索质量和答案可信度。

结语

RAG 的本质是把大模型从「闭卷答题」变成「开卷答题」。

它让模型不必把所有知识都记在参数里，而是可以在回答前查找外部资料。这样既能使用最新和私有知识，也能提供引用来源，降低幻觉风险。

但 RAG 不是银弹。真正可靠的 RAG 系统，需要同时做好：

  • 文档解析
  • 知识切分
  • 向量与关键词检索
  • rerank
  • 上下文组装
  • 忠实生成
  • 权限控制
  • 质量评估

如果把这些环节做好，RAG 就不只是一个「给模型塞资料」的小技巧，而是一套可以把大模型接入真实知识体系的工程方法。
